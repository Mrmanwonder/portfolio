# 07_Projects.md

# Systems & Endurance

## Chapter 03 --- Creation

### Projects Experience Specification

Version 1.0

> The Projects chapter is the emotional and visual centerpiece of the
> portfolio. Visitors should experience each project as its own
> miniature exhibition rather than reading a collection of project
> cards.

------------------------------------------------------------------------

# 1. Narrative Purpose

This chapter answers the question:

**"What can this person actually build?"**

Projects are presented as evidence of systems thinking, design maturity,
and technical execution.

------------------------------------------------------------------------

# 2. Story Structure

Each project follows the same cinematic rhythm:

1.  Arrival
2.  Environment reveal
3.  Device emergence
4.  Product exploration
5.  Technical deep dive
6.  Results
7.  Transition to next project

No project should appear abruptly.

------------------------------------------------------------------------

# 3. Scene Layout

The chapter is pinned during scroll.

A full-viewport showcase panel fills the screen with generous margins.

Within the panel:

-   Background artwork
-   Desktop mockup
-   Tablet (optional)
-   Phone mockup
-   Floating technology icons
-   Project title
-   Short description
-   Metrics strip
-   Call-to-action links

------------------------------------------------------------------------

# 4. Device Choreography

The showcase panel rises from below.

Devices emerge sequentially:

Desktop: - Largest - Appears first

Tablet: - Rotates slightly into place

Phone: - Slides upward with a subtle delay

Every device casts soft shadows and exhibits layered parallax based on
pointer movement.

------------------------------------------------------------------------

# 5. Camera

The visitor does not scroll past projects.

Instead, the camera advances through them.

Rules:

-   Slow forward dolly
-   Slight perspective shift
-   No sudden zooms
-   Smooth transitions between projects

------------------------------------------------------------------------

# 6. Background Identity

Each project owns a distinct visual language while remaining part of the
overall portfolio.

Examples:

AXON: - Electric blues - Data particles - Grid overlays

OPTYX: - Dark interface - OCR overlays - Scanning beams

Study Platform: - Warm whites - Educational diagrams - Animated graphs

------------------------------------------------------------------------

# 7. Screenshot Behaviour

Screenshots are never static.

Permitted interactions:

-   subtle scrolling within mockups
-   hover reveals
-   animated dashboards
-   cursor demonstrations
-   loading states

Avoid looping videos unless necessary.

------------------------------------------------------------------------

# 8. Technical Metadata

Each project presents:

-   Technologies
-   Duration
-   Team size
-   Architecture summary
-   Key challenge
-   Outcome

Display this information using editorial typography instead of badges
wherever possible.

------------------------------------------------------------------------

# 9. Scroll Mapping

0--15% Environment appears.

15--35% Devices emerge.

35--60% Typography enters.

60--80% Interactive exploration.

80--100% Camera departs toward the next project.

------------------------------------------------------------------------

# 10. Motion Language

Movement hierarchy:

Environment

↓

Devices

↓

Typography

↓

Micro details

↓

Particles

Nothing should animate simultaneously without purpose.

------------------------------------------------------------------------

# 11. Performance

Only preload the next project.

Dispose of heavy assets from previous projects once fully exited.

Large images should stream progressively.

Maintain 60 FPS.

------------------------------------------------------------------------

# 12. Accessibility

Reduced Motion:

-   Devices fade into place.
-   Interactive previews become static.
-   Camera remains fixed.
-   All information remains accessible.

------------------------------------------------------------------------

# 13. Acceptance Criteria

The visitor should remember:

-   the visual identity of each project,
-   the engineering behind it,
-   and at least one unique interaction.

The Projects chapter should feel like walking through a modern product
design exhibition rather than browsing a portfolio gallery.
